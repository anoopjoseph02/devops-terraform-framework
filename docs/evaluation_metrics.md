# Evaluation Metrics Framework

This document defines the measurable outcomes for the dissertation evaluation chapter,
addressing the reviewer requirement: *"The project should define measurable outcomes such
as deployment time reduction, failure rate improvement, resource optimisation, or cost savings."*

---

## 1. Primary quantitative metrics

| Metric | Measurement method | Baseline (manual) | Target |
|--------|-------------------|--------------------|--------|
| Provisioning time | Mean CD pipeline duration (deploy_metrics.jsonl) | 120 min [R1] | ≤ 12 min |
| Provisioning time (P95) | 95th percentile CD duration | ~180 min | ≤ 20 min |
| Error rate | Failed runs / total runs | 18% [R1] | ≤ 5% |
| Security issues (HIGH/CRITICAL) | Checkov findings per deployment | 3.2 [R1] | 0 HIGH |
| Security issues (MEDIUM) | Checkov MEDIUM findings | ~5 per deploy | ≤ 2 |
| Cost estimate accuracy | Infracost estimate vs. actual Azure bill | 60% [R1] | ≥ 85% |
| Recommendation acceptance rate | Auto-applied / total recommendations | N/A | ≥ 70% |
| Output consistency | Identical output for identical input across 10 runs | ~55% [R1] | 100% |
| Anomaly detection precision | TP / (TP + FP) on labelled test set | N/A | ≥ 0.80 [R7] |
| Anomaly detection recall | TP / (TP + FN) on labelled test set | N/A | ≥ 0.75 [R7] |
| Predictive scaling accuracy | MAE of node count recommendation | N/A | ≤ 1 node |

---

## 2. Secondary qualitative metrics

- Developer experience: Likert 1–5 survey on framework usability (≥ 4.0 target)
- Code review turnaround: time from PR open to merge (framework vs. manual)
- Prompt-to-plan latency: time from JSON input to `terraform plan` output

---

## 3. Data collection

All primary metrics are collected automatically:

```
reports/
├── deploy_metrics.jsonl        # appended by scripts/capture_metrics.sh after every CD run
├── checkov_summary.json        # generated by metrics-collector.yml weekly
├── recommendations.md          # generated by scripts/ai_recommender.py per run
├── anomaly_report.md           # generated by scripts/anomaly_detector.py post-CD
└── evaluation_report.md        # generated by scripts/evaluate_framework.py
```

To generate the full evaluation report at any time:

```bash
python scripts/evaluate_framework.py
```

## 4. Evaluation protocol (testing phase: 15 Mar – 15 Apr 2026)

### Phase A — Baseline measurement (5 deployments, manual)
1. Provision the same `aks_config.json` spec manually using `az` CLI + hand-written Terraform.
2. Record time, errors, and security issues for each run.
3. Store results in `reports/manual_baseline.csv`.

### Phase B — AI framework measurement (15+ deployments)
1. Trigger the AI generator pipeline with the same spec.
2. deploy_metrics.jsonl accumulates automatically.
3. After 15 runs: `python scripts/evaluate_framework.py`

### Phase C — Anomaly detection validation
1. Inject 3 synthetic anomalous deployments (large resource count, abnormal duration).
2. Verify anomaly_detector.py flags all 3 with precision ≥ 0.80.

### Phase D — Predictive scaling validation
1. Run predictive_scaler.py with 7 days of synthetic CPU data.
2. Compare recommended node count with optimal count derived from CPU peak analysis.
3. Report MAE.
